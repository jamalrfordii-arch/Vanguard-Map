## 1. Point Cloud Rendering in Three.js

### 1a. Potree — The Gold Standard for Large Point Clouds
**URL:** https://github.com/potree/potree


Potree is a free open-source WebGL based point cloud renderer for large point clouds.
 
It is based on the TU Wien Scanopy project.
 
Three.js is the WebGL 3D rendering library on which Potree is built.


**How LOD works:** 
At the heart of Potree's working process lies its Modified Nested Octree data structure. This technique enables Potree to efficiently render and interact with massive point cloud datasets in real time. The Modified Nested Octree divides the point cloud data into hierarchical levels, organizing it into a tree-like structure. Each level of the octree corresponds to varying levels of detail.


**Rendering technique (from Schütz thesis):** 
Points are rendered as squares in WebGL. Circles are obtained by discarding fragments. `gl_PointCoord` stores the normalized coordinates of a square with the origin at the top left. Coordinates are mapped to the distance from the center of the square. Fragments with a distance larger than 1 are discarded.


**EDL shader integration:** 
The EDL shader was adapted from the CloudCompare source code!


### 1b. Three-Loader (Potree for Three.js)
**URL:** https://github.com/pnext/three-loader


The point cloud comes with a material which can be customized directly. Here we just set the size of the points. It updates the visibility of the octree nodes based on the camera frustum and it triggers any loads/unloads which are necessary to keep the number of visible points in check.


### 1c. Three.js `Points` + `PointsMaterial` (Built-in)

Three.js gives us Points 3D object that is internally rendered with flag `gl.POINTS`. The Points class expects two parameters, Material and Geometry.


**Key limitation:** 
Default `THREE.PointsMaterial` does not support lighting (the points are visible with or without adding lights to the scene).
 This is critical for Vanguard1 — you'll need custom shaders or the CSM library.

### 1d. Practical Performance for Large Clouds
**URL:** https://beefed.ai/en/real-time-large-scale-point-clouds


Rendering a billion points in a browser is a systems problem more than a graphics problem: you must treat a point cloud as a streaming, hierarchical dataset with node-local compression, not as a single giant vertex buffer. Done right, you can deliver smooth navigation, accurate measurements, and sub-second picks by combining preprocessing (quantization & tiling), an octree LOD traversal using a screen-space error, GPU-side decoding, and a small, targeted interaction pipeline.


**Data compression tip:** 
A `uint16` per axis is 6 bytes vs 12 bytes for `float32` — that's a 50% reduction before compression. Decode on the GPU using the node's min and span uniforms.

## 2. Advanced Shading for Points (Custom ShaderMaterial)

### 2a. THREE-CustomShaderMaterial (CSM) ⭐ KEY RESOURCE
**URL:** https://github.com/FarazzShaikh/THREE-CustomShaderMaterial

This is the **best way** to add custom GLSL to standard Three.js materials (including PointsMaterial):


`csm_PointSize` is a `float`, direct equivalent of `gl_PointSize`, only available in PointsMaterial. `csm_DiffuseColor` is a custom diffuse color in the Fragment Shader — base material's shading will be applied to this color.


CSM lets you **extend** `MeshPhysicalMaterial` or `PointsMaterial` while keeping Three.js's built-in PBR lighting pipeline. For Vanguard1's tactical points, you can write custom vertex displacement + fragment coloring and still inherit the lighting model.

### 2b. ShaderMaterial Directly
**URL:** https://gist.github.com/duhaime/822f81a578b47d6fd0be523e6f7efc78


We can create our own custom materials in Three.js by writing our own shaders and passing them into a ShaderMaterial, which we can then use in our scene.


**Approach:** 
Attributes only affect the vertex shader; they are applied to geometry. In Three.js they must be applied to BufferGeometry and we create a Float32Array the size of the number of vertices. Then for each vertex we can assign a value to our array.

## 3. Eye-Dome Lighting (EDL) — Depth Perception without Normals ⭐

**URL (Kitware article):** https://www.kitware.com/eye-dome-lighting-a-non-photorealistic-shading-technique/


Eye-Dome Lighting (EDL) is a non-photorealistic, image-based shading technique designed to improve depth perception in scientific visualization images. It relies on efficient post-processing passes implemented on the GPU with GLSL shaders in order to achieve interactive rendering. Solely projected depth information is required to compute the shading function, which is then applied to the colored scene image. EDL can therefore be applied to any kind of data regardless of their geometrical features.



EDL was developed by Christian Boucheny during his Ph.D.


**How it works conceptually:** 
A 3D scene is first projected using an off-screen OpenGL rendering. The resulting color and depth images are stored in two 2D textures. The depth image is then used to calculate the shading by applying the EDL algorithm.


**Implementation reference (Unity shader for point clouds):**
https://github.com/SFraissTU/BA_PointCloud/blob/master/PointCloudRenderer/Assets/Resources/Shaders/EDLShader.shader (referenced in Godot proposals)

**For Vanguard1:** EDL is perfect for a tactical map point cloud — it gives edge/depth perception with zero normals, just a depth buffer pass. Implement as a post-processing pass in Three.js using `EffectComposer`.

## 5. Terrain Texture Splatting

### 5a. Advanced Height-Based Splatting
**URL:** https://www.gamedeveloper.com/programming/advanced-terrain-texture-splatting


One of the most common ways of terrain texturing is blending multiple tiled layers. Each layer has an opacity map which defines extent of texture presence on the terrain. The method works by applying an opacity map to the higher levels, revealing the layers underneath.


The key improvement: 
Stones look evenly soiled by sand, but in real world it doesn't happen. Sand doesn't stick to stones, instead it falls down and fills cracks between them, leaving tops of stones pure. As we want sand to be "fallen down" between cobble-stones, for each texture we need the depth map.


### 5b. Nathan Pointer's Three.js Terrain Rendering Guide ⭐
**URL:** https://nathanpointer.com/blog/landscapes


We should be able to get around this problem by mixing and repeating multiple smaller textures, one for each terrain type (grass, mud, rock etc). Texture splatting works by taking multiple textures and associating them with a color channel in another texture called a splat texture. One pixel in the splat texture gets rendered out as say a 100x100 image. If a pixel in the splat is pure red then we render the area as mud and if its pure green then render it grass.


**Normal blending (Reoriented Normal Mapping):** The article references Stephen Hill's technique at https://blog.selfshadow.com/publications/blending-in-detail/ for blending macro terrain normals with tiled detail normals.

## 6. GPU Terrain Rendering — Geometry Clipmaps

### NVIDIA GPU Gems 2 (Hoppe & Asirvatham)
**URL:** https://developer.nvidia.com/gpugems/gpugems2/part-i-geometric-complexity/chapter-2-terrain-rendering-using-gpu-based-geometry


The geometry clipmap is a level-of-detail structure for rendering terrains. It caches terrain geometry in a set of nested regular grids, which are incrementally shifted as the viewer moves. The grid structure provides simplicity of data structures, smooth visual transitions, steady rendering rate, graceful degradation, efficient compression, and runtime detail synthesis.



By processing terrain geometry as a set of images, we can perform nearly all computations on the GPU itself, thereby reducing CPU load. The technique allows interactive flight over a 20-billion-sample grid of the United States stored in just 355 MB of memory, at around 90 frames per second.

Color correction is a common part of the gaming industry now, from making blue nights and golden sunrises, to increasing contrast, applying hueshifting. Curves are the best way to perform common tasks such as lowering and raising the brightness, adding contrast, HDR, etc.


### 8c. LUT-Based Color Grading (2D texture workaround for WebGL)

The basic idea of color grading is to use such a color cube, but with altered colors, as a 3D lookup table.



This 2D texture is basically emulating a 3D texture, so we can think of it as a series of slices. If the shader tries to lookup a color that falls near the "slices" borders, it will interpolate between two slices, producing wrong results.
 Use half-texel offsets to fix this.

## 9. Normal Estimation for Point Clouds

### Academic Papers (for offline preprocessing)
- **"Fast and Accurate Normal Estimation via Patch Stitching"** — https://arxiv.org/pdf/2103.16066 — 
Presents an effective normal estimation method adopting multi-patch stitching for unstructured point clouds.

- **"Weighted Point Cloud Normal Estimation"** — https://arxiv.org/pdf/2305.04007 — 
Develops a novel weighted normal regression technique that predicts point-wise weights from local point patches for robust, feature-preserving normal regression.

- **Conventional PCA approach:** 
Conventional methods such as Principal Component Analysis (PCA) show limited capability in handling noise or complex point cloud surfaces.


### Terrain-Specific (LiDAR to Mesh)
**ACM paper:** https://dl.acm.org/doi/10.1145/3611314.3615901 — 
Presents a pipeline for reconstructing terrain models using LiDAR point cloud data, which aims to preserve landform features. A normal estimation method leverages the information from the LiDAR scans.


### Real-time GPU Normal Estimation
**Paper:** https://arxiv.org/pdf/2504.15121 — 
Constructs a surface normal estimator that is both fast and accurate, producing a dense, oriented point cloud as the final output. Validated using simulated environments and real-world stereo images, demonstrating significant improvements in real-time performance when implemented on a GPU. Upon acceptance, the shader source code will be made publicly available.

## 🎯 Recommended Architecture for Vanguard1

Based on all findings, here's a suggested rendering pipeline:

1. **Terrain mesh** → Heightmap-displaced `PlaneGeometry` with `ShaderMaterial` using texture splatting (depth-based blending per the GameDeveloper article)
2. **Point cloud overlay** → `THREE.Points` with custom `ShaderMaterial` (or CSM extending `PointsMaterial`) for per-point color, size attenuation, and fake lighting via normal estimation
3. **EDL post-processing** → Render points to an offscreen FBO, compute depth-based edge shading in a fullscreen quad pass
4. **Color grading** → Final LUT-based post-processing pass using 2D texture emulating 3D cube
5. **LYGIA shader functions** → Import PBR lighting, tonemapping, and color utilities as `#include`s
6. **Future-proof** → Target `WebGPURenderer` (r171+) with WebGL2 fallback; leverage compute shaders for LOD traversal and normal estimation
