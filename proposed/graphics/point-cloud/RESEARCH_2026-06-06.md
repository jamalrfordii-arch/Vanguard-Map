## 1. Point Cloud Rendering in Three.js

### Core Architecture

Three.js gives us `Points` 3D object that is internally rendered with flag `gl.POINTS`. The `Points` class expects two parameters, Material and Geometry.
 For full control, 
using `BufferGeometry` takes more time but gives full control over everything geometry-related.


### Potree / three-loader (Large-Scale Point Clouds)
- **GitHub:** https://github.com/pnext/three-loader
- 
Potree updates the visibility of the octree nodes based on the camera frustum and triggers loads/unloads to keep the number of visible points in check.

- **Potree main repo:** https://github.com/potree/potree
- 
A recent (Feb 2026) article covers rendering million-point LiDAR clouds in the browser with Three.js and Potree.


### Custom ShaderMaterial for Points
- **Gist (Points + ShaderMaterial):** https://gist.github.com/duhaime/822f81a578b47d6fd0be523e6f7efc78
- 
The default `THREE.PointsMaterial` does not support lighting (points are visible with or without lights). A custom `ShaderMaterial` is needed for lit point clouds.

- **Observable notebook:** https://observablehq.com/@vicapow/three-js-point-custom-shader

### THREE-CustomShaderMaterial (CSM)
- **GitHub:** https://github.com/FarazzShaikh/THREE-CustomShaderMaterial
- 
CSM provides output variables like `csm_PointSize` (only available in PointsMaterial) and `csm_DiffuseColor` where the base material's shading will be applied to this color.
 This lets you extend `MeshPhysicalMaterial` or `PointsMaterial` with custom GLSL while retaining built-in PBR lighting.

## 2. Eye-Dome Lighting (EDL) — Critical for Point Clouds

EDL is the **single most impactful** technique for point-cloud depth perception without normals.

- 
EDL is a non-realistic rendering technique used to improve depth perception. The only information needed for EDL is a depth map. EDL works by looking at the depth of adjacent pixels in screen space and darkening the pixels which exhibit a sudden change of depth in their neighborhood. The bigger the local difference in depth values is, the darker the color is.

- 
Plain rendered point clouds result in flat shading; enabling Eye Dome Lighting gives a slight perception of depth; adding also Ambient Occlusion enhances the effect even further.

- **Three.js EDL Pull Request:** https://github.com/mrdoob/three.js/pull/14016 — 
This should be implemented as `EDLPass.js` to be used with `EffectComposer`.

- **Potree's EDL shader** was originally 
adapted from the CloudCompare source
, developed by Christian Boucheny.
- 
The technology was first proposed by Christian Boucheny in 2009 and became more popular due to the increasing power of the GPU.

## 3. Ambient Occlusion for Point Clouds (SSAO/GLSL)

### SSAO Fragment Shader Approach
- **Tutorial:** https://kindatechnical.com/computer-graphics/lesson-31-ambient-occlusion.html
- The SSAO approach samples a hemisphere around each fragment's normal and checks depth buffer occlusion. 
Real implementations must handle sample distribution carefully and account for depth discontinuities to avoid halo artifacts.


### EasyPBR Paper (Point Cloud PBR + EDL + AO)
- **Paper (arXiv):** https://arxiv.org/pdf/2012.03325
- 
In point cloud rendering, PBR and IBL cannot be applied due to the lack of normal information. Instead, they rely on Eye Dome Lighting.
 The paper demonstrates combining EDL + SSAO for the best point cloud depth perception.

### Easy3D Library (Reference Implementation)
- **GitHub:** https://github.com/LiangliangNan/Easy3D
- 
Includes rendering techniques like point/line imposters, ambient occlusion (SSAO), hard shadow, soft shadow (PCSS), eye-dome lighting for rendering point clouds without normal information, and transparency.

## 4. Terrain Texture Splatting

### Advanced Height-Based Splatting
- **Article (GameDeveloper.com):** https://www.gamedeveloper.com/programming/advanced-terrain-texture-splatting
- 
One of the most common ways of terrain texturing is blending multiple tiled layers. Each layer has an opacity map which defines extent of texture presence on the terrain. The method works by applying an opacity map to the higher levels, revealing the layers underneath.


### Three.js Terrain Splatting Implementation
- **Blog (nathanpointer.com):** https://nathanpointer.com/blog/landscapes
- 
Texture splatting works by taking multiple textures and associating them with a color channel in another texture called a splat texture. One pixel in the splat texture gets rendered out as say a 100x100 image, making the splat texture significantly smaller than a diffuse texture of the same level of detail.

- Includes a **normal blending** technique: 
The author extended the three.js standard material with custom utilities specific to terrains.
 References Stephen Hill's normal blending method from https://blog.selfshadow.com/publications/blending-in-detail/

### Heightmap Terrain in Three.js
- **Tutorial:** https://techblog.geekyants.com/recreating-real-world-terrain-with-react-threejs-and-webgl-shaders
- 
The vertex shader uses the heightmap to shape terrain, and the fragment shader applies textures using a `ShaderMaterial`.


### THREE.Terrain (Procedural Generator)
- **GitHub:** https://github.com/IceCreamYou/THREE.Terrain
- Supports Diamond-Square heightmap generation, mesh scattering for foliage, and heightmap export/import.

### Three.js Journey — Procedural Terrain Shader
- **Course:** https://threejs-journey.com/lessons/procedural-terrain-shader
- 
Transforms a plane into a procedural terrain with mountains, water, beaches, and plateaus using shaders.

## 5. Color Grading (GLSL Post-Processing)

### LUT-Based Color Grading
- **Code Lab:** https://codelab.wordpress.com/2012/10/22/simple-color-grading-for-games/
- 
Use a 3D texture as a look-up table to map RGB colors. The GLSL shader is essentially: `gl_FragColor = vec4(texture3D(lut, col).xyz, 1.0);`
 The brilliant part: serialize the identity LUT into the screenshot, let an artist grade it in Photoshop, and the LUT is automatically produced.
- **Kosmonaut's blog:** https://kosmonautblog.wordpress.com/2017/04/26/color-grading-correction/
  - 
Going for LUT16 or LUT32 is the right choice, as quality is pretty much on par.


### GLSL Color Correction Shader Library

Includes curves, brightness/contrast, HSL adjustments — common color correction tools ported to GLSL.


### Trevor Van Hoof — Color Grading Shader

## 6. LYGIA Shader Library (Reusable GLSL Functions)

- **GitHub:** https://github.com/patriciogonzalezvivo/lygia
- **Website:** https://lygia.xyz/
- **npm:** `lygia`
- 
Includes `color/` with general color operations: `luma()`, `saturation()`, blend modes, palettes, color space conversion, and tonemaps.

- 
It's a granular and multi-language (GLSL, HLSL, Metal, WGSL, WEGL and CUDA) shader library designed for performance and flexibility.

- 
Most of the code is GLSL and HLSL, but is slowly extending to WGSL, CUDA and Metal.

- **Examples repo:** https://github.com/patriciogonzalezvivo/lygia_examples — includes `color_brightnessContrast`, `color_lut`, `color_dither`, filters, and more.

you can resolve dependencies online via `resolveLygia()` or use bundler plugins (vite-glsl, esbuild-glsl).

## 7. Normal Estimation for Point Clouds

### Screen-Space Normal Generation (dFdx/dFdy)
- **Blog (Enki Software):** https://www.enkisoftware.com/devlogpost-20150131-1-Normal-generation-in-the-pixel-shader
- 
This outputs the world space position to the pixel shader, which then calculates the screen space derivatives to get the face normal.
 However, 
24 bits of mantissa isn't always enough to represent `dFdx(world_pos)` to sufficient precision.


### PCA-Based Normal Estimation
- 
A typical example is based on PCA, a method that calculates each point's normal by computing eigenvectors from the covariance matrix based on neighbouring points.

- 
While simple, it tends to blur details and is fragile on noisy point clouds.


### PCL Library (Preprocessing)
- 
The PCL (Point Cloud Library) estimates normals by fitting a best plane to a small subsection of the clouds.
 Best done as a preprocessing step.

### Academic Papers
- **"Fast and Accurate Normal Estimation for Point Cloud via Patch Stitching"** (arXiv:2103.16066) — 
proposes a fast, accurate framework whose efficiency is not affected by network scale, using adaptive local feature aggregation.

- **"Robust Normal Estimation for 3D LiDAR Point Clouds in Urban Environments"** (Sensors, 2019) — https://www.ncbi.nlm.nih.gov/pmc/articles/PMC6427512/

## 8. Point Cloud Surface Splatting (High-Quality Rendering)

### WebGL EWA Splatter
- **Live Demo:** https://www.willusher.io/webgl-ewa-splatter/
- **GitHub:** https://github.com/Twinklebear/webgl-ewa-splatter
- 
An EWA surface splatter renderer in WebGL implementing "Object Space EWA Surface Splatting" and "High-Quality Point-Based Rendering on Modern GPUs," also using deferred shading for splatting.


### WebSurfels
- **GitHub:** https://github.com/nbasargin/WebSurfels
- 
WebSurfels is a library for high-quality point cloud rendering on the Web. The implementation uses EWA splatting and renders normal-aligned splats to reconstruct the sampled surfaces.


### Compute Shader Point Rasterizer
- **GitHub:** https://github.com/m-schuetz/compute_rasterizer
- 
The basic idea is to spawn a compute shader that transforms points to screen space, encodes depth and color into a single 64-bit integer, and uses `atomicMin` to compute the closest point for each pixel.


### 3D Gaussian Splatting (WebGL)
- **GitHub:** https://github.com/antimatter15/splat
- 
A WebGL implementation of a real-time renderer for 3D Gaussian Splatting. As it is essentially an extension of rendering point clouds, it can be done very efficiently on ordinary graphics hardware.

- **Kishimisu's WebGL implementation:** https://github.com/kishimisu/Gaussian-Splatting-WebGL

## 9. Real-Time Terrain Point Cloud Rendering (Academic)

### Key Paper: "Real-Time Interpolated Rendering of Terrain Point Cloud Data"
- **PMC:** https://pmc.ncbi.nlm.nih.gov/articles/PMC9824316/
- 
Most real-time terrain point cloud rendering techniques do not address the empty space between points. This work proposes an alternative approach that tries to fill it with appropriate values.

- 
The main contribution is a novel adaptive real-time point cloud rendering technique that identifies which points are on the frontmost surfaces using a depth map and filters out the distant points.

- 
Implemented in WebGL 2.0, which allows hardware-accelerated 2D and 3D rendering in browsers.


### GPU Terrain Rendering (Tessellation)
- 
Hybrid CPU-GPU coupled solutions maintain a coarse terrain on CPU via quad tree, while a fine version is produced on the GPU using tessellation shaders.

- 
Geometry clipmaps algorithms allow the entire terrain to be rendered through only one draw call by improving grid structure and index organization.

## 10. Three.js Version Updates & WebGPU/TSL (Future Path)

### Three.js r166 Release
- **GitHub:** https://github.com/mrdoob/three.js/releases/tag/r166
- 
r166 added support for instanced rendering with sorting, frustum culling in `BatchedMesh`.


### Current State: Three.js r184
- 
Three.js is currently at r184.


### WebGPU Renderer + TSL (Three Shading Language)
- 
WebGPURenderer comes with a node-based material system. TSL allows developers to write shader code with JavaScript in a platform-independent manner, transpiled to WGSL or GLSL.

- 
New effects like SSGI, SSS, and better DoF are exclusively available for the new renderer.

- 
Since Three.js r171 (September 2025), WebGPU is production-ready with zero-config imports.

- 
TSL provides a JavaScript-based shader authoring system that compiles to both WGSL (WebGPU) and GLSL (WebGL), letting you maintain a single codebase.

- **Migration guide:** https://www.utsubo.com/blog/webgpu-threejs-migration-guide
- **TSL Field Guide:** https://blog.maximeheckel.com/posts/field-guide-to-tsl-and-webgpu/
- 
The guide covers building simple materials using TSL's node system, and leveraging compute shaders for instanced meshes, particles, or post-processing effects.


### ⚠️ Migration Note

Custom materials based on `ShaderMaterial`, `RawShaderMaterial` and `onBeforeCompile()` are not supported in WebGPURenderer. They must be ported to node materials and TSL.
