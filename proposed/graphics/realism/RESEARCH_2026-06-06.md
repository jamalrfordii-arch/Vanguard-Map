## 1. 🌍 Atmospheric Scattering

### `@takram/three-atmosphere` (★ Best-in-Class)
📦 **NPM:** `npm i @takram/three-atmosphere`
🔗 **GitHub:** https://github.com/takram-design-engineering/three-geospatial


A Three.js and R3F (React Three Fiber) implementation of Eric Bruneton's Precomputed Atmospheric Scattering.
 
This library is part of a project to prototype the rendering aspect of a Web GIS engine.


Key features:
- 
Packages include atmosphere (Precomputed Atmospheric Scattering), clouds (geospatial volumetric clouds), core, and effects collections.

- 
Suitable for large-scale scenes, but supports only Lambertian BRDF.

- 
Currently developed using GLSL. It does not use node-based TSL yet, and WebGPU is not supported, but both are planned.

- 
An optional LUT texture contains only higher-order (N ≥ 2) scattering terms. By using this information, it can attenuate only single scattering in the shadowed segments of rays. This texture is not required unless you have shadow length information, or you use the clouds package with the lightShafts option enabled.

## 2. 🌅 God Rays / Volumetric Light Shafts

### `Erkaman/glsl-godrays` (★ Recommended)
🔗 **GitHub:** https://github.com/Erkaman/glsl-godrays
🎮 **Demo:** http://erkaman.github.io/glsl-godrays/


In order to implement the effect, no less than three rendering passes will have to be done: Pass 1: Render all geometry that could occlude the light source as black. Normally render light source. And render all the above to a texture called the "occlusion texture". Note that this texture does not have to be exactly the size of the screen, but it can be smaller. And by making it smaller, lots of performance can be gained. Pass 2: Render everything normally, to the default framebuffer.


### GPU Gems 3 — Reference Paper
🔗 https://developer.nvidia.com/gpugems/gpugems3/part-ii-light-and-shadows/chapter-13-volumetric-light-scattering-post-process


In our approach, we apply a per-pixel post-processing operation that requires no preprocessing or other scene setup, and which allows for detailed light shafts in animating scenes of arbitrary complexity.
 The key parameters are exposure, decay, density, and weight, each passed as a shader uniform.

### Three.js Fake Volumetric Cone Approach
🔗 https://threejsdemos.com/demos/lighting/godrays


Fake Volumetric Lighting: simulates god rays using a translucent cone mesh with additive blending. ConeGeometry creates light shaft geometry from point light source. Additive Blending: overlapping rays add together for bright concentration effects. Depth Write Disabled: prevents cone from blocking other transparent objects.


### Three.js PathTracing Renderer (God Rays included)
🔗 **GitHub:** https://github.com/erichlof/THREE.js-PathTracing-Renderer


Light Shaft Demo demonstrates the rendering of volumetric light shafts (a.k.a. 'God rays') which occur when light sources are blocked everywhere except for the lit volume.


### Volumetric Light Scattering in Three.js (Article)
🔗 https://medium.com/@andrew_b_berg/volumetric-light-scattering-in-three-js-6e1850680a41


The implementation follows Kenny Mitchell's Volumetric Light Scattering as a Post-Process in GPU Gems 3. The general technique: render the scene with the light source as white and all occluding objects as black to a texture, apply a parameterized radial blur with a shader, and additively blend over the normally lit scene.

### GLSL Foam Techniques

Inigo Quilez's approach: put foam in the top of waves by adding random white color fetched from a "foam texture" based on the height and slope of the wave mesh. You can add a time offset to the fetching coordinates aligned to the wind direction, to let the foam "slide" over the water surface.



The main idea for foam generation in the water surface rendering literature is the application of precalculated foam-texture. The choppiness of waves is evaluated, and on the places where it exceeds a specific level, foam-texture is blended to the final color.


**Key Reference:** Inigo Quilez's Simple Water article — https://iquilezles.org/articles/simplewater/

## 4. 🔲 Screen Space Ambient Occlusion (SSAO)

### N8AO (★ Top Pick)
🔗 **GitHub:** https://github.com/N8python/n8ao
📦 **NPM:** `npm i n8ao`


N8AO is an efficient and visually pleasing implementation of SSAO (Screen Space Ambient Occlusion) that emphasizes temporal stability and artistic control.



Simply add N8AOPass to your EffectComposer — it replaces RenderPass. The effect should work out of the box with any vanilla three.js scene, and as long as the depth buffer is being written to, generate ambient occlusion. Use of SMAAPass (as hardware antialiasing does NOT work with ambient occlusion) is recommended.


Key parameters:
- 
aoRadius controls the radius/size of the ambient occlusion in world units. Should be set to how far you want the occlusion to extend from a given object.

- 
Color of the ambient occlusion can be changed to any color to offer a crude approximation of global illumination. Recommended in scenes where bounced light has a uniform "color", for instance a scene lit by a blue sky.

- 
N8AOPass is compatible with WebGL 2.0 (WebGL 1 is not supported).


### `Rabbid76/three-js-ao-pass` — Unified AO Pass
🔗 **GitHub:** https://github.com/Rabbid76/three-js-ao-pass


Includes implementations of SAO, N8AO, HBAO, and GTAO all in one pass, with links to reference papers for each algorithm.

## 5. 🌑 Shadow Mapping

### Three.js Built-in Shadow System
🔗 **Docs:** https://threejs.org/docs/#api/en/renderers/WebGLRenderer.shadowMap
🔗 **Examples:** https://threejs.org/examples/webgl_shadowmap.html


The shadow rendering system in Three.js's WebGL renderer covers shadow map generation, depth material selection, render target management, and different shadow filtering techniques (PCF, VSM).



The shadow mapping system operates as a separate rendering pass that executes before the main scene render. It generates depth textures from the perspective of each shadow-casting light, which are then sampled during the main render pass to determine shadowed fragments.


Available shadow types:
- `BasicShadowMap` — No filtering
- `PCFShadowMap` — Percentage-Closer Filtering
- `PCFSoftShadowMap` — Softer PCF
- 
VSM (Variance Shadow Maps) store depth moments and use Chebyshev's inequality for smooth shadow filtering.


### Cascaded Shadow Maps (CSM) — For Large Terrains (★ Critical for Vanguard1)
🔗 **GitHub:** https://github.com/StrandedKitty/three-csm
📦 **NPM:** `npm i three-csm`
🔗 **Official Three.js CSM Docs:** https://threejs.org/docs/pages/CSM.html
🔗 **Official Example:** https://threejs.org/examples/webgl_shadowmap_csm.html


This approach provides higher resolution of shadows near the camera and lower resolution far away by using several shadow maps. CSMs are usually used for shadows cast by the sun over a large terrain.



An implementation of Cascade Shadow Maps (CSM). This module can only be used with WebGLRenderer. When using WebGPURenderer, use CSMShadowNode instead. CSM is an addon, and must be imported explicitly.


TypeScript variant: https://github.com/alexnil/three-csm-typescript

## 7. 🌫️ Atmospheric Haze / Distance Fog

### WebGL Fundamentals — Fog Tutorial
🔗 https://webglfundamentals.org/webgl/lessons/webgl-fog.html


The basic technique: use depth/distance from the camera in your shaders to make the color more or less the fog color. `gl_FragColor = mix(originalColor, fogColor, fogAmount);` where fogAmount is a value from 0 to 1.


Three fog types:
1. **Linear fog:** uses `smoothstep(fogNear, fogFar, distance)`
2. **Exponential fog:** 
Fog apparently works exponentially — it gets thicker with the square of the distance from the viewer.

3. **Exponential squared:** sharper falloff

### Height-Based Fog (Inigo Quilez method)
🔗 **GLSL Implementation:** https://github.com/jwagner/terrain/blob/master/shaders/atmosphere.glsl


Uses `exp((-rayOrigin.y*vFalloff)*fogDensity)` formula for height-dependent fog, based on Inigo Quilez's fog article.


**IQ Reference:** https://iquilezles.org/articles/fog/

### Dunes Terrain Fog Example (WebGL)
🔗 https://dev.to/keaukraine/rendering-dunes-terrain-in-webgl-30k2


The next important thing for a realistic result is atmospheric fog. Uses simple linear fog in the vertex shader. Fog range is controlled by two uniforms — fogStartDistance and fogDistance. Fragment shader applies fog color based on this varying. Fog color is adjusted for far terrain edges to blend with sky texture. And the sky texture is also edited to have a distant haze of the same fog color.


### Three.js Built-in Fog
```js
scene.fog = new THREE.Fog(0xcccccc, near, far);         // Linear
scene.fog = new THREE.FogExp2(0xcccccc, density);        // Exponential
```

## 8. 💧 Water Surface Caustics

### Evan Wallace's WebGL Water (★ Seminal Demo)
🔗 https://madebyevan.com/webgl-water
🔗 **Article:** https://medium.com/@evanwallace/rendering-realtime-caustics-in-webgl-2a99a29a0b2c


The vertex shader positions vertices by tracing a ray along the sunlight direction, refracting it through the surface, and intersecting it with the ground geometry. The same technique can be used for computing caustics due to reflection off the top of the water's surface. The demo renders caustics to a texture that can be draped over the scene.


### Martin Renou — Real-Time Water Caustics (Three.js + WebGL)
🔗 https://medium.com/@martinRenou/real-time-rendering-of-water-caustics-59cda1d74aa


Caustics are patterns of light that occur when light is refracted and reflected from a surface. In order to get stable 60fps, they compute entirely using shaders written in GLSL. The approach: compute refracted rays at the water surface, compute where those rays hit the environment, compute caustics intensity by checking where rays converge.



Presents an attempt for generalizing caustics computation in real-time using WebGL and ThreeJS.


### `pabennett/WaterCaustics`
🔗 **GitHub:** https://github.com/pabennett/WaterCaustics


Emulates the light patterns known as water caustics typically seen on the bottom of a swimming pool. Investigates various methods including FBO ping-pong for shallow water ripples and Tessendorf's FFT synthesis technique.
