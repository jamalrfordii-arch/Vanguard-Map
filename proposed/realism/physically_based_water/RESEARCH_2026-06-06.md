## 2. Key GLSL Techniques Required

### A. FFT-Based Wave Spectrum Generation (JONSWAP)

The mathematical foundation uses the JONSWAP spectrum. 
The JONSWAP spectrum is for developing seas, where the waves still have the potential to grow and evolve as a constant wind blows over them. It is defined by: S_JONSWAP(ω) = (αg²/ω⁵) · γʳ · exp[-(5/4)(ωₚ/ω)⁴]



The main idea behind the FFT ocean simulation is that the ocean surface can be represented as a sum of sinusoidal waves. Not just 10 or 20 waves, but thousands of them! The solution is to use an Inverse Fourier Transform: from a given complex wave spectrum that specifies the amplitude and phase of waves in all directions, we can directly compute the sum without computing each wave individually.


**GLSL snippet — JONSWAP Spectrum Initialization:**
```glsl
// JONSWAP spectrum S(ω) for initial wave field
float jonswap(float omega, float omega_p, float alpha, float gamma) {
    float sigma = omega <= omega_p ? 0.07 : 0.09;
    float r = exp(-pow(omega - omega_p, 2.0) / (2.0 * sigma * sigma * omega_p * omega_p));
    float pm = (alpha * 9.81 * 9.81 / pow(omega, 5.0)) * exp(-1.25 * pow(omega_p / omega, 4.0));
    return pm * pow(gamma, r);
}
```

### B. GPU IFFT via Ping-Pong Render Targets (WebGL2 Path)

Since WebGL2 has no native compute shaders, the standard approach uses `GPUComputationRenderer`:


GPUComputationRenderer uses the concept of variables. These variables are RGBA float textures that hold 4 floats for each compute element (texel). Each variable has a fragment shader that defines the computation. The renderer has actually two render targets per variable, to make ping-pong.


```javascript
import { GPUComputationRenderer } from 'three/addons/misc/GPUComputationRenderer.js';

const gpuCompute = new GPUComputationRenderer(256, 256, renderer);
const spectrum0 = gpuCompute.createTexture();
// Fill spectrum0 with JONSWAP-initialized complex amplitudes...
const spectrumVar = gpuCompute.addVariable('textureSpectrum', spectrumFragShader, spectrum0);
// Add butterfly FFT passes...
gpuCompute.init();

// In animation loop:
gpuCompute.compute();
const heightMap = gpuCompute.getCurrentRenderTarget(spectrumVar).texture;
```


It is a helper class that lets you create data textures that store computations via self-defined fragment shaders into 4 floating point numbers (xyzw) in each texel — 32-bit per channel, 16 bytes per texel. A data texture defined in GPUComputationRenderer can depend on its previous frame's result to calculate the next; you can even set up several data textures with multiple dependencies.


### C. Jacobian-Based Foam Detection


At wave crests, the turbulence tends to cause a more diffuse reflection, increasing overall brightness and "whiteness" of the water. By looking at the gradient of the horizontal displacement map, we can detect where the crests occur and use this to pass a turbulence factor to the fragment shader.



When the Jacobian factor is close to 0, the water is very "turbulent" and when larger than 1, the water mesh has been "stretched" out. A Jacobian of 1 is the "normal" state. The lower-frequency Jacobian is combined with the normal map in the fragment shader to compute a final "turbulence" factor.


```glsl
// Jacobian foam — computed from displacement gradients
// dDx/dx, dDx/dz, dDz/dx, dDz/dz stored in displacement texture derivatives
float jacobian = (1.0 + dDx_dx) * (1.0 + dDz_dz) - dDx_dz * dDz_dx;
float foam = saturate(-jacobian + foamBias); // negative Jacobian = folding = foam
foam = pow(foam, foamExponent);
```

### D. Depth-Buffer Foam (Shore/Intersection Foam)


The technique utilizes the depth buffer to compare how close the water surface is to the objects penetrating the surface and then uses that value to determine where foam should be rendered.


```glsl
// Depth-based intersection foam
float sceneDepth = texture(depthTexture, screenUV).r;
float waterDepth = gl_FragCoord.z;
float depthDiff = linearizeDepth(sceneDepth) - linearizeDepth(waterDepth);
float intersectionFoam = 1.0 - smoothstep(0.0, foamRange, depthDiff);
```

### E. Subsurface Scattering (Water Glow)


The shader handles proper light interaction: Fresnel reflections, subsurface scattering through wave peaks, caustic patterns on the seafloor, and foam that forms naturally where waves break.


The fast SSS approximation for water (based on the GDC 2011 Colin Barré-Brisebois technique):

```glsl
// Fast water SSS approximation
vec3 H = normalize(lightDir + normal * sssDistortion);
float VdotH = pow(saturate(dot(viewDir, -H)), sssPower);
float sss = (VdotH + sssAmbient) * sssAttenuation;
vec3 sssColor = waterColor * sss * thickness; // thickness from wave height
```

### F. Real-Time Caustics


Evan Wallace created the webgl-water demo, and the most interesting aspect is his approach to rendering caustics. The term "caustic" refers to the pattern that light makes when it is reflected or refracted by a curved surface.



Computing underwater caustics accurately involves simplifying assumptions: for each point we calculate two refract rays, one using a vertical normal and the other using the real normal. We get the caustic value by comparing the two refract rays, then write the caustic value to the caustic map.


```glsl
// Simplified caustics — compare refracted area ratio
vec3 refractedOld = refract(lightDir, vec3(0, 1, 0), eta); // flat surface
vec3 refractedNew = refract(lightDir, surfaceNormal, eta);  // wavy surface
float oldArea = length(dFdx(refractedOld)) * length(dFdy(refractedOld));
float newArea = length(dFdx(refractedNew)) * length(dFdy(refractedNew));
float caustic = oldArea / max(newArea, 0.001); // bright where light converges
```

### G. Normal Map Animation (Dual-Layer Scrolling)


The ocean surface seems wavy even though it is geometrically completely flat, thanks to 2 scrolling different normal maps generated at different scales.


```glsl
// Dual-layer animated normals
vec3 n1 = texture(normalMap, uv * scale1 + time * dir1).xyz * 2.0 - 1.0;
vec3 n2 = texture(normalMap, uv * scale2 + time * dir2).xyz * 2.0 - 1.0;
vec3 blendedNormal = normalize(vec3(n1.xy + n2.xy, n1.z * n2.z));
```

### H. Flow Maps for Directional Water


The special feature is the usage of a flow map. The fragment shader will sample this texture and convert the color values to 2D flow vectors. These are used to distort the normal values in the direction of flow, effectively creating more realistic water effects.

## 3. Performance Characteristics & GPU Cost

### Budget Breakdown for 60fps at 1080p

| Component | GPU Cost (est.) | Resolution | Notes |
|---|---|---|---|
| FFT Spectrum → IFFT (2 cascades) | 1.5–2.5ms | 256×256 per cascade | Ping-pong via GPUComputationRenderer |
| Vertex displacement | 0.3–0.5ms | Clipmap LOD, ~200k verts | Texture fetch in vertex shader |
| Fresnel + PBR shading | 0.5–1.0ms | Per pixel | Schlick approximation |
| Planar reflection | 1.0–2.0ms | Half-res (540p) | Extra draw pass; SSR cheaper |
| Caustics map | 0.3–0.5ms | 512×512 | Rendered once per frame |
| Foam (Jacobian + depth) | 0.2–0.3ms | Per pixel | Texture lookups |
| SSS | 0.1–0.2ms | Per pixel | Cheap dot-product trick |
| **TOTAL** | **~4–7ms** | | Leaves ~10ms for rest of scene |


The FFT algorithm is a clever way to compute the Fourier Transform in O(n log n) time complexity instead of O(n²).



For small spectrum resolutions, you can probably get away with a CPU-side FFT using multithreading, but for larger resolutions, you will need to use compute shaders.
 On WebGL2, the GPUComputationRenderer serves as the compute shader substitute.


There is very little performance difference in subdivision because once the geometry is uploaded to the GPU there is nothing for the CPU to do but update a time variable once per frame.


**Optimization strategies:**
- Use 128×128 or 256×256 FFT resolution (not 512×512) for WebGL2
- Half-resolution reflections/refractions
- 
Compile-time quality levels control resolution and feature availability. Disabled features are excluded from the shader entirely — no wasted GPU cycles.

- LOD clipmap geometry for infinite ocean without vertex pressure

## 4. Three.js r184-Specific APIs

### TSL (Three Shading Language) — Now Stable in r184


Three.js r184 ships TSL (Three Shading Language) as a first-class shading API. Instead of writing GLSL strings, you compose a node graph in plain JavaScript and assign it to material.colorNode or material.positionNode. The renderer lowers the graph to GLSL for WebGLRenderer and to WGSL for WebGPURenderer at compile time — one source, two backends.



TSL — Three Shading Language — is a JavaScript node-graph API introduced as stable in r184.


This is a game-changer for Vanguard1: you write one water shader in TSL and it automatically compiles to GLSL for WebGL2 and WGSL for WebGPU.

### TSL Water Material Pattern (r184)

```javascript
import * as THREE from 'three/webgpu';
import { 
  uniform, texture, positionLocal, normalLocal,
  sin, time, vec3, vec4, float, mix, 
  Fn, MeshPhysicalNodeMaterial
} from 'three/tsl';

const waterMaterial = new MeshPhysicalNodeMaterial({
  transmission: 0.9,
  roughness: 0.1,
  ior: 1.333,
  thickness: 2.0,
});

// Vertex displacement via FFT heightmap
const heightMap = uniform(texture(fftHeightTexture));
const displacement = Fn(() => {
  const uv = positionLocal.xz.mul(0.01);
  const height = heightMap.sample(uv).r;
  return positionLocal.add(vec3(0, height, 0));
});
waterMaterial.positionNode = displacement();

// Animated normal perturbation
const waterNormal = Fn(() => {
  const t = time.mul(0.05);
  const n1 = texture(normalMapTex, positionLocal.xz.mul(0.1).add(t));
  const n2 = texture(normalMapTex, positionLocal.xz.mul(0.3).add(t.mul(-0.7)));
  return normalLocal.add(n1.xyz.add(n2.xyz).mul(0.5));
});
waterMaterial.normalNode = waterNormal();
```

### WebGL2 Backend via WebGLNodesHandler


A TSL node graph assigned to material.colorNode or material.positionNode compiles to GLSL for WebGLRenderer and to WGSL for WebGPURenderer. Set `glRenderer.setNodesHandler(new WebGLNodesHandler())` to enable the WebGL path. One source, two backends.


### GPUComputationRenderer (WebGL2 GPGPU)


GPUComputationRenderer is an addon, and must be imported explicitly: `import { GPUComputationRenderer } from 'three/addons/misc/GPUComputationRenderer.js';`


This is the primary tool for FFT on WebGL2 — it emulates compute shaders through fragment-shader ping-pong rendering into float textures.
