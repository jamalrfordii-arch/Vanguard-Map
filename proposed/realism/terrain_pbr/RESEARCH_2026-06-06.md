## 2. Key GLSL Techniques Needed

### A. Triplanar Texture Mapping

This is essential for cliff faces and steep terrain where UV-based texturing produces stretching artifacts.


The main algorithm: compute the vertex's normal vector and check what the larger component of the normal is, out of x, y and z. If x is the larger component, use the geometry z coordinate as the texture coordinate s, and y as t. If z is the larger component, use x as s, and y as t.



When using triplanar mapping, you sample maps using three different projections. This triples the amount of texture sampling in the shader. To keep this manageable, aim to minimize the amount of samples per projection. You can do this by storing multiple surface properties in a single map.


**Performance note:** 
You might even want to combine triplanar mapping with texture splatting, but that's expensive because it would require a lot more texture sampling.


### B. Height-Based Splat Blending (Depth-Aware)

Standard linear alpha blending between terrain layers produces unrealistic "smeared" transitions. The advanced technique uses per-layer height maps stored in the alpha channel.


This texture splatting algorithm allows to create more natural terrain. One of the most common ways of terrain texturing is blending multiple tiled layers. Each layer has an opacity map which defines extent of texture presence on the terrain. The method works by applying an opacity map to the higher levels, revealing the layers underneath.


The key insight from the Game Developer article: instead of linear interpolation, compare heights from the alpha channel and use the "taller" texture, with a smoothing depth to avoid pixel artifacts. This lets sand fill cracks between cobblestones naturally.

### C. Anti-Tiling / Stochastic Texturing


One way to prevent the visual repetition of the texture is to assign a random offset and orientation to each tile of the repetition. You can determine which tile the current sample is in, then create pseudo-random values for the tile, and use these to offset and re-orient the texture. Re-orientation can be as simple as mirroring in x or y or rotating. This produces a non-repeating pattern over the whole surface.


**Reference:** Inigo Quilez's technique at https://iquilezles.org/articles/texturerepetition/


Texture tiling results in obvious texture repetition. Inigo Quilez has a great article on multiple ways of fixing this. The technique essentially boils down to using a random noise texture and some math to more organically place and blend the texture seamlessly.


### D. Macro/Micro Normal Blending (UDN / Reoriented Normal Mapping)


World creator exports a normal map for the whole terrain but also a normal for the rock texture tiled across the landscape. Ideally the scene would benefit from both the macroscopic cliff face normal as well as the more nuanced bumps and cracks of tile level normals. Unfortunately, if you just add the normal textures together they get muddy. Stephen Hill has a good summary at https://blog.selfshadow.com/publications/blending-in-detail/


The recommended GLSL function for UDN blending (from the nathanpointer.com landscape article):

```glsl
vec4 blend_normals(vec4 n1, vec4 n2) {
    vec3 t = n1.xyz * vec3(2, 2, 2) + vec3(-1, -1, 0);
    vec3 u = n2.xyz * vec3(-2, -2, 2) + vec3(1, 1, -1);
    vec3 r = t * dot(t, u) / t.z - u;
    return vec4(r, 1.0) * 0.5 + 0.5;
}
// Chain for 3+ layers:
vec4 blend = blend_normals(macroNormal, microNormal);
vec4 triBlend = blend_normals(blend, detailNormal);
```

### E. Displacement via Heightmap (Vertex Shader)


The displacement map affects the position of the mesh's vertices. Unlike other maps which only affect the light and shade of the material, the displaced vertices can cast shadows, block other objects, and otherwise act as real geometry. The displacement texture is an image where the value of each pixel (white being the highest) is mapped against and repositions the vertices of the mesh.


**Critical requirement:** 
You need at least one subdivision of the plane for every pixel in your height map. If you have less, there will not be an associated vertex for Three.js shader to position.


Best practice: 
Displacement for big shapes, normal maps for micro detail. Combine both for the best results.

## 4. Three.js r184-Specific APIs

### TSL (Three Shading Language) ŌĆö Stable in r184


Three.js r184 ships TSL (Three Shading Language) as a first-class shading API. Instead of writing GLSL strings, you compose a node graph in plain JavaScript and assign it to `material.colorNode` or `material.positionNode`. The renderer lowers the graph to GLSL for WebGLRenderer and to WGSL for WebGPURenderer at compile time.



TSL is a JavaScript node-graph API introduced as stable in r184. You compose shading logic from nodes like `positionLocal`, `time`, `mix`, and `mx_noise_float`, then assign the graph to `material.colorNode` or `positionNode`. The renderer compiles it to GLSL for WebGL and to WGSL for WebGPU. Use TSL when you want WebGPU support, or the built-in PBR lighting pipeline via `MeshStandardNodeMaterial`. Stick with ShaderMaterial for existing GLSL codebases or WebGL-only projects.


### Node Material Support in WebGLRenderer


There's been active discussion (Issue #30185) about adding Node Materials to WebGLRenderer. Previously Node Materials were only supported with WebGPURenderer, which posed issues for community adoption. Many tools developed on WebGLRenderer cannot be ported easily.



Currently `NodeBuilder` has two extended classes: `WGSLNodeBuilder` aimed at WebGPU and `GLSLNodeBuilder` aimed at WebGL2.


### WebGPURenderer with WebGL2 Fallback


The three/webgpu entry point automatically falls back to WebGL 2 if WebGPU isn't available. Global coverage: ~95% of users have WebGPU-capable browsers. The remaining 5% get WebGL 2 fallback automatically.


### Key r184 Renderer Improvements

- 
Implement `WEBGL_multi_draw` fallback
 ŌĆö helps batch terrain chunk draw calls
- 
Support unfilterable float32 StorageTextures
 ŌĆö useful for heightmap data
- 
Introduce `ReadbackBuffer` and reuse `getArrayBufferAsync()` buffers
 ŌĆö useful for GPUŌåÆCPU terrain height queries
- 
Introduce Timeline (Commands Recorder)


### `onBeforeCompile` ŌĆö The Proven WebGL2 Approach


You can use the `onBeforeCompile` property on the material. If you assign a function to it, this function will be called with the shader options as the first parameter right before being compiled. We now have access to the `vertexShader`, the `fragmentShader`, and the `uniforms` and we can modify these.



This is possible with a callback on the Material class called `.onBeforeCompile`. This is a very powerful and amazing feature. It is invoked by WebGLRenderer before two important steps ŌĆö parsing and compiling.

### Decision: WebGL2 `onBeforeCompile` vs TSL

For **maximum control and proven stability** on WebGL2 today, use `MeshStandardMaterial` + `onBeforeCompile` to inject your terrain GLSL. This is the approach used by `nickfallon/textureSplat` and by the nathanpointer.com landscape demo.

For **future-proofing**, consider TSL with `MeshStandardNodeMaterial`. 
TSL covers ShaderMaterial (classic GLSL), and Three's 2026 TSL node graph ŌĆö a JavaScript-native alternative that compiles to GLSL for WebGL and WGSL for WebGPU from one source.
 The `@interverse/three-terrain-lod` package already uses TSL for its DefaultTerrainMaterial.

### Implementation Skeleton (onBeforeCompile approach)

```javascript
import * as THREE from 'three';

// Terrain chunk geometry (from @interverse/three-terrain-lod or custom)
const chunkGeo = new THREE.PlaneGeometry(256, 256, 128, 128);

// PBR base textures (per-layer, packed)
const loader = new THREE.TextureLoader();
const splatMap  = loader.load('terrain_splat.png');
const layer0    = { albedo: loader.load('grass_albedo.jpg'),
                    normal: loader.load('grass_normal.jpg'),
                    orm:    loader.load('grass_orm.jpg') };
// ... layers 1-3 similarly

// Create the material
const terrainMat = new THREE.MeshStandardMaterial({
    map:         layer0.albedo,
    normalMap:   layer0.normal,
    roughness:   1.0,
    metalness:   0.0,
    envMapIntensity: 1.0,
});

terrainMat.onBeforeCompile = (shader) => {
    // Inject uniforms
    shader.uniforms.u_splatMap   = { value: splatMap };
    shader.uniforms.u_tileScale  = { value: 32.0 };
    // ... add all layer textures as uniforms

    // Inject declarations into fragment shader
    shader.fragmentShader = shader.fragmentShader.replace(
        '#include <map_fragment>',
        /* glsl */ `
        // ŌöĆŌöĆ SPLAT-BASED PBR TERRAIN ŌöĆŌöĆ
        vec4 splat = texture2D(u_splatMap, vUv);
        vec2 tiledUV = vUv * u_tileScale;

        // Sample each layer
        vec4 c0 = texture2D(u_layer0Albedo, tiledUV);
        vec4 c1 = texture2D(u_layer1Albedo, tiledUV);
        vec4 c2 = texture2D(u_layer2Albedo, tiledUV);
        vec4 c3 = texture2D(u_layer3Albedo, tiledUV);

        // Height-aware blending (depth in alpha channel)
        // ... advanced blending logic here

        // Weighted blend
        vec4 albedo = c0 * splat.r + c1 * splat.g
                    + c2 * splat.b + c3 * splat.a;
        diffuseColor = albedo;
        `
    );
};
```

### Key GLSL Module: Height-Aware Depth Blending

```glsl
// Per-pixel height-aware blending (from gamedeveloper.com technique)
vec3 heightBlend(vec4 tex1, float a1, vec4 tex2, float a2) {
    float depth = 0.2; // transition sharpness
    float ma = max(tex1.a + a1, tex2.a + a2) - depth;
    float b1 = max(tex1.a + a1 - ma, 0.0);
    float b2 = max(tex2.a + a2 - ma, 0.0);
    return (tex1.rgb * b1 + tex2.rgb * b2) / (b1 + b2);
}
```

### Key GLSL Module: Conditional Triplanar

```glsl
vec4 triplanarSample(sampler2D tex, vec3 worldPos, vec3 worldNormal,
                     float tileScale) {
    vec3 blend = abs(worldNormal);
    blend = pow(blend, vec3(4.0));  // sharpen
    blend /= dot(blend, vec3(1.0));

    vec4 xProj = texture2D(tex, worldPos.yz * tileScale);
    vec4 yProj = texture2D(tex, worldPos.xz * tileScale);
    vec4 zProj = texture2D(tex, worldPos.xy * tileScale);

    return xProj * blend.x + yProj * blend.y + zProj * blend.z;
}

// In main(): only use triplanar on steep slopes
float slopeFactor = 1.0 - worldNormal.y; // 0 = flat, 1 = vertical
if (slopeFactor > 0.4) {
    albedo = mix(albedo, triplanarSample(...), smoothstep(0.4, 0.7, slopeFactor));
}
```
