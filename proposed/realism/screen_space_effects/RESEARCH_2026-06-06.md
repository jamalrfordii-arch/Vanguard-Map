## 1. AMBIENT OCCLUSION — Best Available Implementations

### A. **N8AO** (Recommended for WebGL2 / Production) ⭐

**GitHub:** https://github.com/N8python/n8ao
**NPM:** `npm i n8ao`
**Live Demo:** https://n8programs.com/n8ao/


N8AO is a screen-space ambient occlusion (SSAO) post-processing library for Three.js applications. The library provides an efficient implementation optimized for temporal stability and artist control, designed to add depth perception to 3D scenes by darkening areas where geometry occludes ambient light.


**Key features:**
- 
It works with logarithmic depth buffers and orthographic cameras, supports materials with custom vertex displacement and alpha clipping, and automatically detects the presence of these things so you do not need to deal with user-side configuration.

- 
The color of the AO can be changed to any color to offer a crude approximation of global illumination.
 This is relevant for Vanguard1 — you can tint AO to simulate environment bounce light.
- 
The repository is structured to support two parallel integration paths: native Three.js postprocessing and the pmndrs/postprocessing library. Both implementations share the same shader core and configuration system, ensuring consistent visual results regardless of framework choice.


**Setup (vanilla Three.js EffectComposer):**
```js
import { N8AOPass } from "n8ao";
import { EffectComposer } from "three/addons/postprocessing/EffectComposer.js";
import { SMAAPass } from "three/addons/postprocessing/SMAAPass.js";

const composer = new EffectComposer(renderer);
const n8aopass = new N8AOPass(scene, camera, width, height);
composer.addPass(n8aopass);
composer.addPass(new SMAAPass(width, height)); // SMAA recommended
```

**Setup (pmndrs/postprocessing):**
```js
import { N8AOPostPass } from "n8ao";
import { EffectComposer, RenderPass } from "postprocessing";

const composer = new EffectComposer(renderer);
composer.addPass(new RenderPass(scene, camera));
const n8aopass = new N8AOPostPass(scene, camera, width, height);
composer.addPass(n8aopass);
```

**Configuration for Vanguard1 (tactical map scale):**
```js
// For a scene that is ~1000 units across (military map)
n8aopass.configuration.aoRadius = 5.0;
n8aopass.configuration.distanceFalloff = 1.0;
n8aopass.configuration.intensity = 3.0;
n8aopass.configuration.color = new THREE.Color(0.05, 0.05, 0.1); // slight blue bounce
n8aopass.configuration.screenSpaceRadius = false;
n8aopass.configuration.halfRes = true; // critical for 60fps
n8aopass.configuration.gammaCorrection = false; // if using OutputPass later

// Quality preset
n8aopass.setQualityMode("Medium"); // or custom:
n8aopass.configuration.aoSamples = 16;
n8aopass.configuration.denoiseSamples = 8;
n8aopass.configuration.denoiseRadius = 12;
```

**Performance characteristics:**
- 
The half-resolution mode uses depth-aware upscaling by default, and this generally incurs a fixed cost of around 1ms. The AO effect looks horrible without depth-aware upscaling, so it is not recommended to disable it.

- 
You can switch between quality modes (the default is Medium). Generally, half-res mode at "Ultra" quality is slightly slower than full-res mode at "Performance" quality, but produces significantly better results.

- 
You can enable debug mode for performance tracking — the `n8aopass.lastTime` variable will have a value set each frame which represents the time, in milliseconds, the AO took to render.


| Quality Preset | Typical GPU Cost (1080p) | Visual Quality |
|---|---|---|
| Performance | ~0.5–1.0 ms | Minimal AO, may be noisy |
| Low | ~1.0–1.5 ms | Visible but subtle |
| **Medium (default)** | **~1.5–2.5 ms** | **Good for 60fps target** |
| High | ~2.5–4.0 ms | Very clean |
| Ultra | ~4.0–6.0 ms | Best temporal stability |
| **Half-res + Ultra** | **~2.0–3.0 ms** | **Best quality/perf ratio** ⭐ |

### B. **GTAOPass** (Official Three.js Built-in)

**Docs:** https://threejs.org/docs/pages/GTAOPass.html
**Demo:** https://threejs.org/examples/webgl_postprocessing_gtao.html
**Import:** `three/addons/postprocessing/GTAOPass.js`


This is a Ground Truth Ambient Occlusion (GTAO) post-processing effect, a screen-space ambient occlusion technique that approximates indirect lighting by sampling the horizon of the hemisphere around each pixel. The implementation includes ambient occlusion darkening and optional bounce light color bleeding effects.



The GTAO system is a multi-stage post-processing pass that integrates into THREE.js's EffectComposer pipeline. It renders intermediate buffers (depth, normals), performs horizon-based ambient occlusion sampling, and composites the results with optional spatial blur filtering.


**Key GLSL technique — Horizon-Based Sampling:**

The GTAO shader performs horizon-based ambient occlusion sampling using the depth and normal buffers. The shader defines are updated dynamically to trigger recompilation when parameters change.


**Color bounce (pseudo-GI):**

When ENABLE_COLOR_BOUNCE is enabled, the shader samples the scene color at each step and accumulates it based on the surface normal alignment. The accumulated color is stored in the RGB channels of the GTAO buffer and represents light bouncing from nearby surfaces.


**Performance optimization:**

The GTAO buffer can be rendered at a lower resolution (renderTargetScale < 1.0) for performance, while depth and normal buffers remain at full resolution for accurate edge detection during blur. The pass uses a single blue noise texture with four channels, each containing an independent blue noise pattern.


**Setup:**
```js
import { GTAOPass } from "three/addons/postprocessing/GTAOPass.js";
import { OutputPass } from "three/addons/postprocessing/OutputPass.js";

const gtaoPass = new GTAOPass(scene, camera, width, height);
gtaoPass.output = GTAOPass.OUTPUT.Default;
composer.addPass(gtaoPass);
composer.addPass(new OutputPass());

// Tune GTAO parameters
gtaoPass.updateGtaoMaterial({
  radius: 0.5,
  distanceExponent: 2,
  thickness: 10,
  scale: 1,
  samples: 16,       // 8-32 range
  directionOffset: 0,
  stepOffset: 0,
});

// Tune Poisson denoiser
gtaoPass.updatePdMaterial({
  lumaPhi: 10,
  depthPhi: 2,
  normalPhi: 3,
  radius: 4,
  rings: 2,
  samples: 16,
});
```

### C. AO Algorithm Comparison (SSAO vs HBAO vs GTAO)


A useful mental model: "SSAO is a fast hack with visible artefacts; HBAO is a smarter hack that knows about geometry; GTAO is calibrated against an offline reference and is what you want when you can afford it."



On any modern GPU released in the last three or four years, all three are essentially free at 1080p — the choice is about quality and motion stability, not frame rate.



SSAO is the original screen-space algorithm from 2007 — fast but noisy with visible halos. HBAO marches rays through the depth buffer to find the horizon angle and produces smoother results. GTAO is the most recent of the three; it is calibrated to match offline ray-traced AO and is the most stable in motion. All three operate in screen space and have similar cost on modern GPUs.


**Recommendation for Vanguard1:** Use **N8AO** for WebGL2 (production path) — it's the most battle-tested third-party solution. Use **GTAOPass** if you want zero external dependencies. N8AO's color-tinted AO gives you a crude GI approximation for free.

## 2. SCREEN SPACE REFLECTIONS (SSR)

### A. **0beqz/screen-space-reflections** (Best for WebGL2) ⭐

**GitHub:** https://github.com/0beqz/screen-space-reflections
**NPM:** `npm i screen-space-reflections`


Features include: Jittering and blurring reflections to approximate rough reflections; Using three.js' WebGLMultipleRenderTarget (WebGL2 only) to improve performance when rendering scene normals, depth and roughness; Early out cases to compute only possible reflections and boost performance. Introduced Temporal Reprojection to reduce noise for the reflections when moving the camera by reprojecting the last frame's reflections into the current one.


**Setup:**
```js
import { SSREffect } from "screen-space-reflections";
import * as POSTPROCESSING from "postprocessing";

const composer = new POSTPROCESSING.EffectComposer(renderer);
const velocityDepthNormalPass = new VelocityDepthNormalPass(scene, camera);
composer.addPass(velocityDepthNormalPass);

const ssrEffect = new SSREffect(scene, camera, {
  intensity: 1,
  exponent: 1,
  distance: 10,           // max reflection distance
  fade: 0,
  roughnessFade: 1,
  thickness: 10,           // depth buffer thickness
  ior: 1.45,               // index of refraction for fresnel
  maxRoughness: 1,
  maxDepthDifference: 10,
  blend: 0.9,              // temporal blend
  correction: 1,
  correctionRadius: 1,
  blur: 0.5,
  blurKernel: 1,
  blurSharpness: 10,
  jitter: 0.3,             // reduces banding
  jitterRoughness: 0.3,
  steps: 20,               // ray march steps
  refineSteps: 5,          // binary search refinement
  missedRays: true,
  useNormalMap: true,
  useRoughnessMap: true,
  resolutionScale: 0.5,    // half-res for 60fps
  velocityResolutionScale: 0.5,
});

const ssrPass = new POSTPROCESSING.EffectPass(camera, ssrEffect);
composer.addPass(ssrPass);
```

**Key artifact mitigation:**

Since SSR only works with screen-space information, there'll be artifacts when there's no scene information for a reflection ray. This usually happens when another object occludes a reflecting object behind it. To make missing reflections less apparent, use an env-map that can then be used as a fallback when there is no reflection. Ideally use a box-projected env-map.


**Performance notes:**
- `steps: 20` + `refineSteps: 5` at half resolution ≈ 2–4 ms on mid-range GPU
- `resolutionScale: 0.5` is essential for 60fps at 1080p+
- MRT G-buffer rendering (WebGL2) avoids redundant scene renders

### B. **SSRPass** (Official Three.js Built-in)

**Docs:** https://threejs.org/docs/pages/SSRPass.html
**Demo:** https://threejs.org/examples/webgl_postprocessing_ssr.html

## 3. SCREEN SPACE GLOBAL ILLUMINATION (SSGI)

### A. **realism-effects** by 0beqz (Best WebGL2 SSGI) ⭐

**GitHub:** https://github.com/0beqz/realism-effects
**NPM:** `npm i realism-effects`
**Live Demo:** https://realism-effects.vercel.app/


A new version of SSGI (version 2) and realism-effects in general is currently in the works. You can checkout the most recent progress on the v2 branch. It features many performance and quality improvements over the current released version.


This is the closest you'll get to UE5 Lumen (screen space) on WebGL2. 
0beqz's graphics projects for three.js include SSGI (real-time global illumination on the web), SSR (screen-space reflections), and TRAA (cinematic anti-aliasing).


**Full pipeline setup (SSGI + HBAO + TRAA + Motion Blur):**
```js
import * as POSTPROCESSING from "postprocessing";
import {
  SSGIEffect,
  TRAAEffect,
  MotionBlurEffect,
  HBAOEffect,
  VelocityDepthNormalPass,
} from "realism-effects";

const composer = new POSTPROCESSING.EffectComposer(renderer);

// VelocityDepthNormal pass is REQUIRED — shared by SSGI, TRAA, and motion blur
const velocityDepthNormalPass = new VelocityDepthNormalPass(scene, camera);
composer.addPass(velocityDepthNormalPass);

// SSGI (replaces RenderPass — it renders the scene itself)
const ssgiEffect = new SSGIEffect(scene, camera, velocityDepthNormalPass, {
  distance: 5.0,
  thickness: 3.0,
  denoiseIterations: 1,
  denoiseKernel: 2,
  denoiseDiffuse: 10,
  denoiseSpecular: 10,
  radius: 5,
  phi: 0.5,
  lumaPhi: 5,
  depthPhi: 2,
  normalPhi: 50,
  roughnessPhi: 50,
  specularPhi: 50,
  envBlur: 0.5,
  importanceSampling: true,
  steps: 20,
  refineSteps: 4,
  resolutionScale: 0.5,     // Half-res critical for 60fps
  missedRays: false,
});

// HBAO for fine contact shadows
const hbaoEffect = new HBAOEffect(composer, camera, scene);

// TRAA for cinematic anti-aliasing
const traaEffect = new TRAAEffect(scene, camera, velocityDepthNormalPass);

// Motion Blur
const motionBlurEffect = new MotionBlurEffect(velocityDepthNormalPass);

// Combine all into single pass (efficient)
const effectPass = new POSTPROCESSING.EffectPass(
  camera,
  ssgiEffect,
  hbaoEffect,
  traaEffect,
  motionBlurEffect
);
composer.addPass(effectPass);
```


If you use SSGI, then you don't have to use the RenderPass anymore as SSGI does the rendering itself. You can save performance by leaving it out.


**SSGI Performance (estimated at 1080p):**
| Configuration | GPU Cost | Notes |
|---|---|---|
| Full res, 20 steps | 8–15 ms | Too expensive for 60fps |
| **Half res, 16 steps** | **4–7 ms** | **Viable at 60fps with budget** |
| Half res, 8 steps | 2–4 ms | Noisy but fast, needs temporal |

One developer tried the node-based SSGI approach, but found it becomes too expensive for large open-world scenes, and is now exploring alternatives like Global SDF + cone tracing for approximate GI.


### B. **SSGINode** (Official Three.js — WebGPU/TSL only)

**Docs:** https://threejs.org/docs/pages/SSGINode.html


Post processing node for applying Screen Space Global Illumination (SSGI) to a scene. The quality and performance of the effect mainly depend on sliceCount and stepCount. The total number of samples taken per pixel is sliceCount * stepCount * 2.



SSGINode was added as a new node for screen-space global illumination.


**Quality presets:**

Without temporal filtering — Low: sliceCount of 1, stepCount of 12. Medium: sliceCount of 2, stepCount of 8. High: sliceCount of 3, stepCount of 16.



With temporal filtering — Low: sliceCount of 2, stepCount of 6. Medium: sliceCount of 3, stepCount of 8. High: sliceCount of 4, stepCount of 12.


**Important: SSGINode requires WebGPURenderer or WebGPURenderer with WebGL2 fallback. It uses TSL, not GLSL.**

```js
// WebGPU path (SSGINode)
import * as THREE from "three/webgpu";
import { pass, mrt, output, normalView } from "three/tsl";
import { ssgi } from "three/addons/tsl/display/SSGINode.js";

const renderer = new THREE.WebGPURenderer({ antialias: true });
await renderer.init();

const postProcessing = new THREE.PostProcessing(renderer);
const scenePass = pass(scene, camera);
scenePass.setMRT(mrt({ output, normal: normalView }));

const ssgiNode = ssgi(scenePass.getTextureNode("output"), scenePass.getTextureNode("normal"));
postProcessing.outputNode = ssgiNode;
```

## 4. THREE.JS r184-SPECIFIC APIs


r184 is the current latest release of Three.js.
 Key changes relevant to screen space effects:

### r183 changes (foundation for r184):
- 
GTAONode — Optimize render target format; Add basic support for temporal filtering.

- 
SSGINode — New node for screen-space global illumination. SSSNode — Add new node for Screen-Space Shadows.


### r184 migration notes:
- 
The internal AO render target of GTAONode has been optimized. The AO is now only accessible in the r channel. The AO blend on application level should use this formula now: `vec4( scenePassColor.rgb.mul( aoPass.r ), scenePassColor.a )`.

- 
GTAONode computes a more physically correct ambient occlusion. However, the new AO tends to be darker and reaches wider than the previous one. Consider lowering radius and scale as a compensation.


### New Post-Processing Architecture (WebGPURenderer):

The whole system sits on top of TSL (Three.js Shading Language), which lets you describe GPU operations in JavaScript instead of writing raw GLSL shader code. Built-in effects like bloom() and dotScreen() are TSL functions.



The whole thing is WebGPU-native. RenderPipeline is built for WebGPURenderer from the ground up with an automatic WebGL2 fallback, so the same code runs on both.


**MRT for efficient G-Buffer rendering:**
```js
// WebGPURenderer + TSL MRT approach
const scenePass = pass(scene, camera);
scenePass.setMRT(mrt({
  output,
  normal: directionToColor(normalView),
}));
const colorTex = scenePass.getTextureNode("output");
const normalTex = scenePass.getTextureNode("normal");
```

## 5. EMERGING TECHNIQUE: SSILVB (Visibility Bitmask GI)


Recent advances in screen-space global illumination have yielded exceptional improvements to the realism and quality of real-time scenes, even over GTAO. One particular advancement is SSILVB, a screen-space global illumination technique that eases the computational burden by keeping track of the occluded horizon via a bitmask.



The visibility mask extension doesn't change the core algorithm, but rather than calculating a single angle, it uses a bitmask that allows "sectors" to be occluded or not. The authors used an unsigned int as the mask, which gives 32 sectors of occlusion. With this addition, you get not only a more accurate representation of the light or shadow, but also directional information, which allows for more realistic ambient occlusion, and can be used for indirect lighting.
